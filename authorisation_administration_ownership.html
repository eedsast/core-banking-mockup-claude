Authorisation vs Administration — Entity Ownership

Each entity is split across both services — each service owns a different facet · includes service authorisation per organisation and branch

Administration service
Authorisation service
Organisation
Owned here
Legal entity, country, timezone, lifecycle status. Created by Superadmin.
Not here — org is a structural concept, not an identity or access concept
Branches
Owned here
Hierarchy, codes, region, parent branch, status. Linked to organisation.
Not here — branch is a structural concept, not an access concept
Service Access (per Org / Branch)
Service enablement config — owned here
Which microservices are enabled for a given organisation and/or branch. Configured by Superadmin. Stored as a service enablement matrix keyed by org_id + branch_id + service_code.
service_codes[ ]
Service scope enforcement — owned here
At token issuance, resolves the user's org_id + branch_id to embed only the permitted service scopes. Requests to non-enabled services are rejected at token validation.
Users
User profile — owned here
Name, organisation, branch assignment, employment status, contact details. Source of record.
user_id
User credentials — owned here
Identity record, password hash, MFA config, active sessions, issued tokens. Created on user.created event.
Roles
Role assignment — owned here
Which role a specific user holds within a specific org or branch. Business configuration decision.
role_code
Role definitions — owned here
What permissions and scopes each role carries. RBAC rules evaluated at token issuance.
Provisioning flow in sequence
From Superadmin action to a user holding a valid token with embedded permissions and service scopes
1
Superadmin → Administration service
Create Organisation
Legal name, country, timezone, base locale. Organisation receives a unique org_id.
2
Superadmin → Administration service
Create Branch under Organisation
Branch code, region, address, parent branch (if hierarchy). Linked via org_id.
3
Superadmin → Administration service
Configure service access per Organisation / Branch
Superadmin selects which microservices are enabled for this org and optionally overridden per branch. Stored as a service enablement matrix: { org_id, branch_id, service_codes[ ] }. A branch inherits its org's service set unless explicitly restricted.
4
Administration service → Event bus
Publishes service.access.configured event
Payload: { org_id, branch_id, service_codes[ ] }. Authorisation service subscribes and caches the enablement matrix in its own read model — no runtime lookup needed.
5
Superadmin → Administration service
Create User, assign role LOCAL_ADMIN
User profile created with name, email, org and branch binding. Role assignment stored in Administration.
6
Administration service → Event bus
Publishes user.created event
Payload: { user_id, org_id, branch_id, role_code }. Administration is the source — it does not call Authorisation directly.
7
Authorisation service ← Event bus
Creates credential record for user_id
Identity record initialised. Welcome email / password-set flow triggered. Credential store is Authorisation's own — never shared with Administration.
8
Authorisation service ← Event bus
Resolves role_code → permission set → stores in read model
Role definitions are looked up internally. The resolved permission set is stored in Authorisation's own read model so no cross-service call is needed at login time.
9
User → Authorisation service
Authenticates, receives token with embedded scopes and service access
Token contains user_id, org_id, branch_id, role, scopes[], services[]. The services[] claim lists only the microservices enabled for the user's org and branch — self-contained, no further auth calls needed per request.
10
Token → Any downstream service
Permission and service access validated locally from token claims
Each microservice validates the token signature, reads scopes[] for permission checks, and verifies its own service_code is present in services[]. Requests to non-enabled services are rejected with 403 Service Not Enabled. No runtime call back to Authorisation on the hot path.
